üß† Git Mental Model

Git works across three layers:

  • Working Directory: your actual files
  • Index (Staging Area): what will be in your next commit
  • HEAD (Repository): your last committed snapshot

Some examples: - git add moves changes ‚Üí index - git commit moves index ‚Üí HEAD - git reset moves HEAD pointer (soft/mixed/hard)

üîÅ Reverting and Resetting ‚Äì Recap

git revert

  • Creates a new commit that undoes a previous one
  • Safe on shared branches (e.g. main)
git revert <commit-hash>

git reset

  • Moves your branch pointer backward
  • Can rewrite history (dangerous on shared branches!)
  • Options:
    • --soft: Keep changes staged
    • --mixed: Keep changes unstaged
    • --hard: Deletes changes
git reset --soft HEAD~1

üßµ Detached HEAD

  • Happens when you check out a specific commit, not a branch
  • Useful for exploring history
  • If you make changes in this state, they can be lost unless you make a branch

üß™ Demo: Detached HEAD + Branch

  1. In the terminal or Git pane:
git log --oneline
  1. Copy a commit hash from earlier and run:
git checkout <commit-hash>

You‚Äôre now in detached HEAD!

üõü Rescue Your Work

If you want to keep changes made in detached HEAD:

git switch -c recovered-work

‚úÖ This creates a new branch from that point!

‚úçÔ∏è git commit --amend

Use this when you:

  • Made a typo in the last commit message
  • Forgot to include a file
  • Want to slightly edit the last commit

üß™ Demo: Amending a Commit

  1. Make a small commit
  2. Realize you forgot something
  3. Stage the missing file, then run:
git commit --amend

üéØ You can edit the message or leave it the same

‚ö†Ô∏è Avoid --amend if you‚Äôve already pushed the commit

üîç git reflog

  • Shows every move of HEAD (including things Git log forgets)
  • Useful to recover commits after a reset or checkout
git reflog

üß™ Activity: Oops, I reset too far!

  1. Make a commit
  2. Reset to a previous state:
git reset --hard HEAD~1
  1. Run git reflog to find the lost commit
  2. Use:
git checkout <hash>

Then:

üéâ You‚Äôve recovered ‚Äúlost‚Äù work!

üóëÔ∏è git rm: Remove Files from the Repository

  • Use git rm to delete a tracked file from both your working directory and Git‚Äôs index.
  • Git will stage the deletion automatically.
git rm filename.txt
git commit -m "Remove filename.txt"

‚úÖ After this commit, the file will no longer exist in your working directory or the repository.

‚ùó Why Not Just Delete the File?

  • If you delete a file in RStudio or your file browser:
    • Git will show it as ‚Äúdeleted: filename.txt‚Äù
    • But you still need to git add the deletion
# After deleting manually:
git status         # Shows deletion
git add filename.txt
git commit -m "Remove file"

‚úÖ git rm does both steps at once ‚Äî it‚Äôs safer and cleaner

üí° Recovering a Deleted File

If you haven‚Äôt committed the deletion yet:

git restore filename.txt

If you already committed the deletion:

git checkout HEAD~1 -- filename.txt

‚úÖ Or use git reflog to find the commit where the file still existed

üßæ git rm --cached: Keep the File, Remove It from Git

  • Use this when you‚Äôve added a file to Git by mistake, but you don‚Äôt want to delete it from your computer
  • Git stops tracking it ‚Äî but the file stays in your working directory
git rm --cached data.csv
git commit -m "Stop tracking data.csv"

‚úÖ The file will still be on disk, but not in the next commit

üß† Common Use Case

You committed a large data file by accident

  1. Add the filename to .gitignore
  2. Run:
git rm --cached data.csv
git commit -m "Remove data.csv from tracking"
  1. Push your changes
  2. The file is now ignored and won‚Äôt be re-added

‚úÖ Useful for cleaning up mistakes without losing the file

üßπ git clean

  • Deletes untracked files (not in Git yet)
  • Helpful when switching branches or cleaning clutter
git clean -n   # dry run
git clean -f   # permanently delete

‚ö†Ô∏è Warning

  • git clean is permanent
  • You can‚Äôt undo it with reflog or reset
  • Always use -n first to preview

üß™ Activity: Cleaning Up

  1. In RStudio, create a few new files but don‚Äôt add them to Git
  2. Run:
git status
git clean -n
  1. If it looks right:
git clean -f

‚úÖ Now those untracked files are gone!

üß† When to Use What

Situation Use This
Undo safely on shared branches git revert
Clean up your last commit git commit --amend
Rewind local commit history git reset
Accidentally reset or lost work git reflog
Delete untracked junk git clean

üç• What Is Squashing?

Squashing means combining multiple commits into a single one.

It‚Äôs useful for:

  • Cleaning up messy history
  • Preparing code before pushing or merging
  • Grouping small fixes into one meaningful commit

Squashing doesn‚Äôt change your code ‚Äî just your history!

üß† How Squash Works

Before squashing:

A -- B -- C -- D  (main)
         ‚Üë
       Fix typo

If B, C, and D are all part of one logical change, we can squash:

A -- [B+C+D]  (main)

‚úÖ One clean commit ‚Äî easier to read, review, and share.

üõ†Ô∏è How to Squash Commits

Use interactive rebase to squash:

git rebase -i HEAD~3

You‚Äôll see:

pick abc123 Add feature
pick def456 Fix typo
pick ghi789 Update docs

Change to:

pick abc123 Add feature
squash def456
squash ghi789

Then save + edit the commit message.

‚ö†Ô∏è When (and When Not) to Squash

‚úÖ Good times to squash:

  • Before pushing a feature branch
  • To combine tiny commits (‚Äúfix typo‚Äù, ‚Äúrename var‚Äù, ‚Äúoops‚Äù)

‚ùå Avoid squashing:

  • After you‚Äôve pushed to a shared branch (rewrites history!)
